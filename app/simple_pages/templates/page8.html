SOLID

Single-responsibility Principle (SRP)

SRP states that "A class should have one and only one reason to change, meaning that a class should have only one job." (1)
An example of SRP in the calculator program would be the following:

class Subtraction:
  @staticmethod
  def subtract(value_1, value_2):
    return value_1 - value_2

This example showcases the SRP design principle because it performs only one job which is to subtract two inputs from each other.

Open-closed Principle (OCP)

OCP states that "Objects or entities should be open for extension but closed for modification." (1) In other words, this principle encourages programmers to write classes or entities of code in a certain way so that if new functionality is need it the existing code does not need be modified.

Liskov Substitution Principle

This principle states "that a superclass object should be replaceable with a subclass object without breaking the functionality of the software." (2)

Interface Segregation Principle

This principle states that "A client should never be forced to implement an interface that it doesn’t use, or clients shouldn’t be forced to depend on methods they do not use." (1)

Dependency Inversion Principle

This principle states that "High-level modules, which provide complex logic, should be easily reusable and unaffected by changes in low-level modules, which provide utility features." (3) This can be achieved by using abstraction.
